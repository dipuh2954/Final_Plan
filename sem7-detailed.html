Interview Q&A — write answers out fully, don't just think them
Tue–Sat — Work through these each night, 4–5 questions per session
Q: Explain the difference between StateFlow and SharedFlow.
StateFlow is a hot flow with a single current value (like LiveData). It always has a value, replays the latest to new collectors, and is designed for UI state. SharedFlow is a hot flow with configurable replay and buffer — designed for one-time events like navigation or snackbars. Key interview answer: never use StateFlow for one-time events like "show a toast" because a new subscriber gets the event again even after it was handled.
Q: What is structured concurrency? Why does it matter?
Structured concurrency means every coroutine must be launched within a CoroutineScope, and when the scope is cancelled, all child coroutines are cancelled automatically. No orphaned coroutines can leak. In Android: ViewModelScope is cancelled when the ViewModel is cleared, so all coroutines inside it are automatically cancelled — preventing memory leaks from coroutines holding references to destroyed UI.
Q: What causes unnecessary recompositions in Compose, and how do you fix it?
Recomposition triggers when the inputs to a composable change. Common causes: (1) Passing unstable types (regular classes, List instead of ImmutableList) — fix with @Stable or @Immutable. (2) Reading a State inside a composable that changes too frequently — use derivedStateOf{} to compute a derived value only when its own inputs change. (3) Lambda captures — use remember { derivedStateOf { ... } }.
Q: Explain the Repository pattern and why you use it.
Repository is a single source of truth for a type of data. It abstracts the data sources (network, database, cache) from the ViewModel. The ViewModel says "give me movies" — it doesn't care if they come from Room or Retrofit. The Repository decides: check cache first, if stale fetch from network, update cache, emit from cache. Result: offline-first behaviour, testable ViewModels (mock the repository), and a single place to change data strategy.
Q: What is the difference between launch and async in coroutines?
launch{} starts a coroutine and returns a Job. Used for fire-and-forget operations (saving to DB, sending analytics). async{} starts a coroutine and returns a Deferred<T> — you call .await() to get the result. Used when you need a result or want to run two operations in parallel: val a = async { fetchUser() }; val b = async { fetchPosts() }; val result = Pair(a.await(), b.await()) — both network calls run concurrently.
Q: How does Hilt inject dependencies into a ViewModel?
When you annotate a ViewModel with @HiltViewModel and its constructor with @Inject, Hilt generates a factory for it. When Compose calls hiltViewModel(), it uses this factory to create the ViewModel with all dependencies already injected. The ViewModel itself never calls Hilt.inject() — Hilt creates it. This is constructor injection, and it's why your ViewModel is testable: in tests, you create it manually by passing mock dependencies to the constructor.

"What is the difference between remember and rememberSaveable?" — remember survives recompositions but not rotation. rememberSaveable survives rotation + process death by saving to a Bundle. Use rememberSaveable for anything the user typed or scrolled to. Use remember for derived UI state that can be recomputed.
"Explain Cold Flow vs Hot Flow." — Cold flow: a new flow execution starts for every collector. Like a movie on demand — every viewer starts from the beginning. Hot flow (StateFlow, SharedFlow): one execution, multiple collectors share the same emissions. Like live TV.
"What is the difference between a Service, an IntentService, and a Foreground Service?" — Service: runs on main thread, for short operations. IntentService: deprecated, ran on background thread with a queue. Foreground Service: shows a persistent notification, gets higher priority from the OS, for long-running user-visible operations (music playback, navigation, fitness tracking).

Tue — Design an Instagram-like Feed
Requirements: Show a paginated feed of posts from followed users. Like/unlike. Infinite scroll. Works offline for recently loaded content.
Data model: Post(id, userId, imageUrl, caption, likesCount, createdAt), User(id, name, avatarUrl), LocalFeedItem = Post joined with User. Room table for offline cache.
Architecture decision: Paging 3 with RemoteMediator — fetch from server, store in Room, serve from Room. Room is the single source of truth. When offline, Room serves the cached page. When online, server pages are written to Room and automatically propagate to UI via Flow.
Like operation: Optimistic update — immediately update UI, fire background API call. If API fails: revert UI update and show snackbar. This is what Instagram actually does. Never make the user wait for a network call for a simple like.
Wed — Design WhatsApp-style Chat
Real-time layer: Two options — WebSocket (persistent connection, low latency, your server maintains connections) vs Firebase Firestore (managed real-time, no server maintenance, higher latency, costs money per read). For a startup or demo: Firestore. For a product company's chat: WebSocket with server.
Message states: Sent (1 tick) → Delivered (2 ticks) → Read (2 blue ticks). Store as an enum in Room and Firestore. Update via: sender sets SENT, FCM delivers to recipient's device → app sets DELIVERED, recipient opens chat → app sets READ, emit event back to sender.
Offline message queue: If no network, store message in Room with status PENDING. WorkManager observes PENDING messages and retries when network returns. User sees a clock icon instead of a tick until delivered.
Thu — Design Zomato/Swiggy Order Tracking
Real-time tracking: Delivery partner sends GPS location every 5 seconds via websocket/FCM. Android app receives location updates and animates the rider marker on Google Maps. Use LatLngInterpolator to animate smooth movement between GPS points rather than teleporting.
Order state machine: PLACED → CONFIRMED → PREPARING → OUT_FOR_DELIVERY → DELIVERED. Each state change triggers a push notification. Foreground Service keeps the location tracking alive even when app is backgrounded — critical because users check other apps while waiting for food.
Fri — Design an Image Loading Library (Coil/Glide internals)
This is a favourite interview question: "How would you implement image loading from scratch?" Answer: (1) Check memory LRU cache (fast, in-RAM). (2) Check disk LRU cache (slower, persistent). (3) If neither: fetch from network, decode Bitmap, add to both caches. (4) Show placeholder while loading, fade in on success. (5) Cancel loading when View/Composable is recycled/leaves composition.
Sat — Design Offline-First Architecture (generic)
Framework for any offline-first mobile feature: (1) UI always reads from local database (Room), never directly from network. (2) A sync manager periodically (or on demand) fetches from server and writes to Room. (3) Room Flows propagate changes automatically to UI. (4) Conflict resolution: server wins for shared data (chat, social), local wins for personal data (draft messages, unsent forms). Practice explaining this in 2 minutes.

Read on commute
Read: developer.android.com/jetpack/compose/performance/stability. The Compose compiler marks each type as Stable or Unstable. A composable with only Stable inputs is "skippable" — if inputs haven't changed, the compiler generates code to skip recomposing it entirely. Regular Kotlin classes are marked Unstable by default because the compiler can't guarantee they won't change.
