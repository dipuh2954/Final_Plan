💻 Night Block · 10 PM
Technical round prep — Android depth
Tue — Coroutines interview answers
Write out these 5 answers fully — not bullet points, full paragraphs:
"What is structured concurrency?" — Answer: coroutines are tied to a CoroutineScope. When the scope is cancelled (e.g., ViewModel is cleared), all child coroutines cancel automatically. This prevents the memory leaks that plagued pre-coroutines async code. In Android: viewModelScope is the most common scope, tied to ViewModel lifecycle.
"What is the difference between StateFlow and SharedFlow?" — Answer: StateFlow always has a current value (it's a state holder — like LiveData). SharedFlow doesn't have a current value; it's an event emitter. Use StateFlow for UI state. Use SharedFlow for one-time events like navigation, toast messages, error events that shouldn't replay on resubscription.
"What is Dispatchers.IO vs Main vs Default?" — Answer: IO is optimised for disk/network operations (can run 64+ threads). Main is the Android UI thread — only UI operations. Default is for CPU-intensive work (sorting, computation). Always withContext(Dispatchers.IO) for Room and Retrofit calls, never on Main.
Wed — Compose interview answers
"What triggers recomposition?" — Answer: any State read inside a composable function. When the State value changes, Compose schedules a recomposition of every composable that read that state. Importantly: reading the same State in a parent function recomposes the parent and all children. This is why state hoisting matters.
"What is derivedStateOf and when do you use it?" — Answer: when a state value is computed from another state but changes less frequently. Example: a button is enabled only when a text field is non-empty. Rather than recomposing every keystroke, derivedStateOf only triggers recomposition when the derived result actually changes (empty → non-empty or vice versa).
"What is the difference between remember and rememberSaveable?" — Answer: remember stores value for the current composition lifetime — lost on configuration change (rotation). rememberSaveable stores the value across configuration changes using the SavedStateHandle under the hood. For anything the user typed or selected, use rememberSaveable.
Thu — Architecture interview answers
"Why Clean Architecture?" — Answer: it enforces the dependency rule — inner layers have no knowledge of outer layers. Domain layer has zero Android imports. This means: (1) domain logic can be unit tested without Android framework. (2) Swapping Retrofit for Ktor only changes the data layer. (3) The codebase is modularisable because layers are decoupled.
"Why Hilt over manual DI?" — Answer: at scale, manually wiring 40+ dependencies creates maintenance burden. Hilt generates the boilerplate at compile time (no runtime reflection like Dagger). Scoping is explicit and IDE-navigable. HiltViewModel integrates directly with Compose. The overhead is negligible and the scalability benefit is real.
Fri/Sat — Your project deep-dives
The most common Android technical round question: "Walk me through [your project]." Practice this answer for every project until it flows completely naturally. Time limit: 4 minutes per project. Structure: (1) What problem it solves. (2) Architecture decision and why. (3) The hardest technical challenge you solved. (4) What you'd improve given more time.
For Finance Tracker specifically: be ready to explain why you built Canvas charts instead of using a library (shows initiative, ownership, deep Compose knowledge). Be ready to explain the Biometric flow (ProcessLifecycleOwner, what happens on auth failure). Be ready to explain EncryptedSharedPreferences and why regular SharedPreferences would be wrong for financial data.

W86 System Design Rounds — Mobile Architecture at Scale ▾
The System Design Framework — use this every time Step 1 (2 min): Clarify requirements. "Should this work offline? How many users? Real-time or periodic sync?" Step 2 (5 min): Define the API and data models. Step 3 (10 min): Architecture — client-side layers, caching strategy, background sync. Step 4 (10 min): Deep dive into the hardest component. Step 5 (3 min): Trade-offs — what did you choose and what did you give up? This framework turns a vague question into a structured 30-minute answer.
💻 Night Block · 10 PM — System Design Practice
1 design per night · 40 minutes on paper
Tue — Design: Instagram-style Feed
40 minutes on paper. No phone. Use the framework.
Requirements to clarify: How many posts per feed? Real-time new posts or pull-to-refresh? Video support? Offline browsing?
API: GET /feed?cursor={lastPostId}&limit=20 — cursor-based pagination, not page numbers. Why? Page numbers break when new posts are inserted. Cursor-based is stable.
Client architecture: PagingSource fetches from Retrofit. RemoteMediator caches to Room. Paging 3 serves from Room. Coil handles image loading with LRU memory cache + disk cache. LazyColumn displays with AsyncImage.
Trade-offs you made: Room cache = stale data possible (trade-off: stale vs always-online). No WebSocket = no instant new post notification (trade-off: real-time vs battery). LRU evicts old images (trade-off: memory vs disk).
Wed — Design: Zomato Delivery Tracking
The hard part: real-time driver location updates on a map every 5 seconds. Two approaches: (1) Polling — app requests location every 5 seconds (simple, battery-heavy). (2) WebSocket — server pushes updates (complex, battery-efficient). State your trade-off explicitly.
Offline handling: if connection drops during delivery, show last known location with a "Connection lost" indicator. Store last location in Room so it shows on app reopen.
Push notifications for order status: FCM. Delivery agent app updates Firestore, Cloud Function triggers FCM to customer. Know this chain — it's directly based on your Chat App work.
Thu — Design: Google Pay / UPI Payment App
Security requirements (clarify these first): biometric before every transaction. Transaction history encrypted at rest. Certificate pinning to prevent MITM.
Transaction history: Room with TypeConverter for amounts (store as Long paise, display as ₹ formatted). Encrypted with SQLCipher or per-entry EncryptedSharedPreferences for small data.
Offline handling: UPI cannot work offline (it requires bank server). Show "No internet — transactions require a connection" rather than pretending offline payment is possible. Honesty in constraints is a sign of engineering maturity.
Fri — Design: YouTube-style Video Player
The hard part: background playback. Solution: Foreground Service with MediaSession and MediaController. The service continues playing when the user navigates away. Notification shows playback controls (MediaStyle notification).
Caching: ExoPlayer (or Media3) handles adaptive streaming. Pre-cache next video in queue using ExoPlayer's DownloadManager when on WiFi. Bandwidth-aware: switch from 1080p to 480p automatically when network degrades (HLS adaptive streaming).
Sat — Design your own app (Chat App or Finance Tracker)
Design Finance Tracker as if you're explaining it in an interview. You built it — you know every decision. But can you articulate: why Room over SQLite? Why EncryptedSharedPreferences for settings? Why WorkManager for notifications? Why Canvas charts instead of MPAndroidChart? Each "why" is a trade-off. Practice articulating trade-offs — this is what separates mid-level from senior candidates.

Phase 3 Offers, Negotiation & Decision W88–W92 ▾
W88 First Offer — Do Not Accept Immediately ★ Milestone ▾
The most expensive mistake you can make this week Accepting the first offer the same day it arrives. Companies know they can get a yes out of an excited candidate by creating artificial urgency ("This offer expires in 24 hours"). This is a negotiation tactic. Every legitimate company will give you at least 1 week to review an offer. If a company refuses to give you 5 business days to review, that tells you something important about how they treat employees. Ask for the time. You will almost always get it.
💻 When the offer arrives — do these in order
Same evening. Don't delay any of these steps.
Step 1 — Express genuine excitement, buy time
Call or email back within 2 hours: "Thank you so much — I'm really excited about this offer and this role. Could I have until [date 7 days out] to review everything carefully and give you a thoughtful response? I want to make sure I'm fully aligned before committing." This is always granted. This time is where your negotiation happens.
Step 2 — Accelerate every other active pipeline (same evening)
Email every other company where you are in any active stage (OA done, Technical R1 done, anywhere):
"Hi [Recruiter], I wanted to let you know that I have received an offer from another company with a deadline of [date]. I'm very interested in [Company] and would strongly prefer to explore this opportunity. Is there any way to accelerate the remaining process before that date?"
This message is honest, professional, and effective. It creates real urgency without being manipulative. Expect approximately 30–40% of companies to respond with an accelerated timeline. This is the fastest way to manufacture competing offers.
Step 3 — Research and calculate total compensation
Read the offer letter completely. Note every component: Base salary, Variable pay / performance bonus, Joining bonus, ESOPs (note the vesting schedule — 4-year vest with 1-year cliff is standard), Medical insurance cover, Flexible benefits. The total year-1 number is what you compare, not just base.
Check Levels.fyi and AmbitionBox for this company + role level. Is their offer at, above, or below market? If below market: counter with confidence. If at market: still counter — the worst they say is no.
Step 4 — Determine your counter number
Your counter formula: market rate × 1.1 to 1.2. If the market rate for this role + company is ₹18 LPA and they offered ₹16 LPA, counter at ₹19–20 LPA. If they offered ₹19 LPA (above market), counter at ₹21 LPA. Always counter once, regardless.
Know your three numbers: Target (what you want), Floor (absolute minimum you'd accept), Market (what the data says). Target is what you ask for. Floor is your mental reservation — never share it.

W89 Counter Every Offer — Every Single One, Without Exception ▾
Why you counter even offers you love Negotiating does not make you look greedy. It makes you look like someone who knows their value. Hiring managers expect negotiation — they build a 10–15% buffer into initial offers specifically because they know candidates will counter. By not countering, you leave that buffer on the table permanently. The salary you accept now affects your next raise (usually a % of current), your next job's starting point, and your lifetime earnings. One conversation is worth potentially ₹1–2 LPA annually.
💻 When to counter — and exactly what to say
Counter within 5 days of receiving any offer
The counter conversation — phone or email
Phone call (preferred — harder to say no to a voice): "Hi [Recruiter], thank you again for the offer — I'm genuinely excited about this role and the team. I've had a chance to review everything and do some market research. Based on my published apps, the Google AAD certification, and the scope of this role, I was hoping we could discuss bringing the base salary to ₹[your target number]. Is there flexibility there?"
Then: stop talking. The silence after you state your number is the most important 10 seconds of the negotiation. Do not fill it. Do not justify further. Do not say "but I understand if that's not possible." Just wait for their response.
Email counter (if phone not possible)
"Hi [Recruiter], thank you for the offer — I'm very excited about joining [Company] and contributing to the Android team. After reviewing the package and researching current market rates on Levels.fyi and AmbitionBox, I'd like to propose a base salary of ₹[target]. This reflects the value of my published Android apps, Google AAD certification, and 2+ years of Kotlin/Compose development. I'm confident I can add immediate value and hope we can align on this. Please let me know your thoughts."
Counter only once — then decide
Counter once. If they come back with an improved but not full number: say "I appreciate the flexibility. I'll review the updated offer and get back to you by [date]." Then decide based on the full picture, not just base salary. A ₹2 LPA raise at a great company beats ₹4 LPA more at a miserable one.
If they say "this is our best and final": "Thank you for being direct. I'll review the full package and get back to you by [date]." Now make your decision based on the total offer and your alternatives. Do not counter a second time after "best and final."
